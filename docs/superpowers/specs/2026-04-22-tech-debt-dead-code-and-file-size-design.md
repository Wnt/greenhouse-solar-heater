# Tech Debt: Dead Code & File-Size/SRP — Design

**Date:** 2026-04-22
**Status:** Draft — awaiting user review
**Scope:** `playground/` + `server/` only. `shelly/` deferred.

## 1. Context & motivation

All code work on this repository is done by agentic AI tools (primarily Claude, plus Copilot and Codex). The single biggest friction for that workflow, in order:

1. **Oversized files** dilute an agent's working context and make edits less reliable — the bigger the file, the less of it fits comfortably next to the surrounding code an agent also needs to read.
2. **Dead code** wastes agent context and can mislead reasoning ("why doesn't this path work?" when the path is actually unreachable).
3. **Drift** — rules that live only as prose in `CLAUDE.md` tend to erode. Mechanical CI checks don't.

This is the first slice of a broader tech-debt effort. It targets only the two highest-value dimensions (dead code, file size / single-responsibility) in the two most active codebases (`playground/`, `server/`). `shelly/` has real platform constraints that change the shape of the work and is deferred to a future slice. Broader architecture changes (async-pattern migration, cross-subsystem duplication, dependency upgrades, repo hygiene) are explicitly deferred.

Prior art: `specs/017-architecture-code-review/` from 2026-04-02 did a full review and implemented its P1 (security) items. Its P2/P3 items around duplication and async patterns are not addressed in this slice and are intentionally left for a later one.

## 2. Goals & non-goals

### Goals

1. Bring every source `.js` file under `playground/js/**` and `server/**` under a strict line-count ceiling so AI agents reason reliably over any single file.
2. Remove genuinely dead code (unused exports, files, npm deps, unreferenced assets) so agents don't waste context on unreachable paths.
3. Codify both as **CI-enforced rules**, without an allowlist, so future features can't silently rebuild the debt.

### Non-goals

- `shelly/` code. Deferred to a later slice with its own platform-aware design.
- Architectural restructuring beyond what the file-size rule forces. No new abstractions, no new patterns, no "while we're here" refactoring.
- Behavioral changes. Every split is mechanical; the app does exactly what it did before, same public API, same WebSocket protocol.
- P2/P3 items from `specs/017-architecture-code-review/` (code duplication, async patterns). Separate slice.
- Repo hygiene: duplicate-numbered `specs/` directories, stale `.drawio.bkp` files, dependency upgrades. Separate slice.

## 3. Success criteria (merge gate)

The feature branch cannot be merged to `main` until all six of these hold:

1. Every source `.js` under `playground/js/**` and `server/**` is **≤ 600 lines** (raw line count, blanks and comments included).
2. Every test `.js` under `tests/**` is **≤ 1200 lines**.
3. `knip` runs with zero findings — no unused exports, files, or npm dependencies.
4. The unused-assets check runs with zero findings — no file in `playground/assets/**` or non-vendored `playground/public/` is unreferenced.
5. Full `npm test` green.
6. A manual Chrome-DevTools-MCP smoke pass documented in the merge PR description, covering all five playground views (`#status`, `#components`, `#controls`, `#device`, `#settings`) plus login and one override-enter/exit cycle.

Rules 1–4 are enforced by CI. No allowlist — any violation fails the build.

## 4. Rules

### 4.1 File-size budget

- Source `.js` hard cap: **600 lines**.
- Test `.js` hard cap: **1200 lines**.
- Counted: raw lines including blanks and comments. Simpler and less arguable than SLOC.
- Measured over:
  - Include: `playground/js/**/*.js`, `server/**/*.js`, `tests/**/*.js`
  - Exclude: `node_modules/**`, `playground/vendor/**`, `playground/public/**`, `playground/assets/**` (non-JS), `tests/output/**`, `scripts/**`, `shelly/**`

### 4.2 Dead code

- `knip` zero findings across the monorepo.
- Unused-assets check zero findings across `playground/assets/**` and non-vendored `playground/public/`.
- Vendored third-parties (`playground/vendor/js-yaml.mjs`, `playground/public/simplewebauthn-browser.mjs`, `playground/public/qrcode-generator.mjs`, `playground/public/material-symbols.css`, `playground/public/material-symbols-outlined.woff2`) are excluded from dead-code detection by explicit configuration.

### 4.3 Exclusions and why

- `shelly/**`: out-of-slice per scope decision.
- `playground/vendor/**`, `playground/public/**` vendored files: explicit CLAUDE.md rule keeps them vendored; `knip` would otherwise flag unused vendored exports.
- Generated files: `playground/assets/bootstrap-history.json`, `playground/assets/system-topology.*`, icon PNGs regenerated by `scripts/make-icons.mjs`, `playground/assets/liquid-glass-*.png` regenerated by `scripts/generate-liquid-glass.mjs`. These are validated by existing drift tests; excluded from dead-code checks.

## 5. Tooling

Three checks, each a small script, wired into `npm test` / CI. No new heavyweight frameworks.

### 5.1 `scripts/check-file-size.mjs`

Custom Node script. Globs the include set, splits source vs. test by suffix (`*.test.js` → test bucket, everything else → source), fails non-zero if any file exceeds its cap. Output: sorted list with line counts and a `✗` marker on violators. Rules are inline in the script, not in a separate config, to keep them discoverable.

### 5.2 `knip`

Add `knip` as a dev dependency. One `knip.json` at the repo root, two logical workspaces:

- **Server** — entry: `server/server.js`. Catches unused exports, unused files under `server/**`, unused npm dependencies listed in root `package.json`.
- **Playground** — entry points: the module scripts declared in `playground/*.html` via `knip`'s HTML plugin. Catches unused JS files under `playground/js/**` and unused exports.

Known-good entries (dynamically referenced handlers, event-dispatch targets) are enumerated in `knip.json` with a one-line comment per entry explaining why the dynamic reference exists. `ignoreExportsUsedInFile: true` for pragmatic handling of same-file private/public distinction.

Script: `npm run knip`. CI fails on any finding.

### 5.3 `scripts/check-unused-assets.mjs`

Custom Node script. For each candidate file in `playground/assets/**` and non-vendored `playground/public/`, greps the repository (`*.html`, `*.js`, `*.mjs`, `*.css`, `*.json`, `*.ts`) for the bare filename. Zero matches → fail, listing the path.

Indirect references are handled by explicit patterns in the script, each with a comment:

- `manifest.json` icon set referenced by filename.
- Material Symbols woff2 referenced from `material-symbols.css` via `@font-face`.
- Any file listed in `scripts/make-icons.mjs` or `scripts/generate-liquid-glass.mjs` as a generation target (these are produced, not consumed, but ship to production).

### 5.4 CI wiring

Extend `.github/workflows/` test job with:

```yaml
- run: node scripts/check-file-size.mjs
- run: npm run knip
- run: node scripts/check-unused-assets.mjs
```

Local convenience: `npm run lint:debt` runs all three.

No pre-commit hook. CI is sufficient and pre-commit would slow every commit.

### 5.5 Tooling alternatives considered and rejected

- ESLint `max-lines`: ESLint is not currently wired in this repo. Adding it just for one rule is disproportionate.
- `ts-prune` / `depcheck` / `unimported`: superseded by `knip`, which is the actively-maintained consolidator.
- `purgecss`: only one production stylesheet of note (`playground/public/style.css`), small. Manual CSS sweep during the refactor suffices; no CI gate needed for this slice.

## 6. Refactor process

### 6.1 Branch strategy

Single long-lived feature branch (e.g. `debt/dead-code-and-file-size`). Many small commits. Single merge to `main` only once every merge-gate criterion holds.

### 6.2 Ordering within the branch

1. **Tooling, soft mode.** Land `scripts/check-file-size.mjs`, `knip` config, `scripts/check-unused-assets.mjs`, and CI wiring as **warnings** (exit 0 on finding). Produces a baseline report.
2. **Dead-code sweep.** Delete what `knip` and the asset check flag as dead. Two commits: JS dead code, asset dead code. Some files may drop under the cap naturally.
3. **File splits, largest-first.** Order:
   1. `playground/js/main.js` (3556)
   2. `server/server.js` (1101)
   3. `playground/js/auth.js` (759)
   4. `server/auth/webauthn.js` (718)
   5. `server/lib/db.js` (615)
   6. `tests/control-logic.test.js` (2083)
4. **Flip CI gates to hard-fail.** One mechanical commit switching the three scripts to exit non-zero on findings. Safe because every file is already under cap.
5. **Merge to `main`.**

### 6.3 Per-file split protocol

Applied identically to each file over cap.

1. Read the whole file once; identify seams; group functions and state by responsibility.
2. Name target modules *before* moving code. Write the target list into the first commit message for that file.
3. Move in small commits, one extracted module per commit (or per tightly coupled pair). Each commit is a pure behavior-preserving move. Two patterns are allowed, per-extraction choice:
   - **Update call sites in the same commit.** Move the code to a new file, update every importer (or dynamic reference) to point at the new path, delete the original definitions. Preferred when the call-site surface is small.
   - **Re-export shim from the original file.** Move the code; the original file re-exports the moved symbols so existing importers keep working. Preferred when call sites are many. Shims get cleaned up in a later commit on the same branch, once every importer is moved — the merge gate requires zero leftover shims (they'd be flagged by `knip` as re-exports with no remaining local use, or simply left in place as genuine entry re-exports if the HTML loads the file directly).

   No renames unless the new location makes the existing name misleading. No simplifications, no new abstractions.
4. Run `npm test` after every commit. Must stay green. If a move breaks tests, fix or revert in the same commit — no "will fix next commit" debt.
5. Once the original file is under cap, run the Chrome-DevTools-MCP smoke pass (see §6.5) and document in the final commit message for that file.

### 6.4 What is and isn't allowed in-slice

Allowed:

- Pure moves with re-exports.
- Deleting code `knip` / unused-asset check flags as dead.
- Renaming a moved identifier only if retaining the name would mislead a reader at the new location.
- Splitting a test file by logical grouping (one `describe` per file, or one scenario cluster per file).

Not allowed (belongs in a future slice):

- Changing any function signature.
- Reorganizing data structures.
- Replacing callbacks with promises, or any other async-pattern migration.
- Merging duplicated code across subsystems.
- Changing the public API of any server endpoint, WebSocket message, or MQTT topic.
- Reformatting unrelated code.

### 6.5 Smoke-test protocol (Chrome DevTools MCP)

Run after each file's splits are complete, not after every intermediate commit.

Flow (driven via MCP):

1. Start local server (`node server/server.js` without `AUTH_ENABLED`).
2. Open `http://localhost:3000/#status`; verify page renders and a sensor value shows non-stale.
3. Navigate `#components`, `#controls`, `#device`, `#settings` in turn; verify each view renders without console errors.
4. Trigger override-enter (e.g. force greenhouse heating for 60 s), verify the UI reflects the override, then override-exit; verify return to normal.
5. Capture a page screenshot and console-log dump; attach condensed output to the commit message for that file.

For `server.js` splits, the same UI-level flow exercises the routes and WS handlers; no separate server-only smoke is needed.

### 6.6 Commit message discipline

Every commit on this branch: `debt: <area>: <what>`. Examples:

- `debt: tooling: add check-file-size script in warn mode`
- `debt: playground: extract live-history fetching from main.js`
- `debt: server: split WebSocket handlers out of server.js`
- `debt: tooling: flip CI gates to hard-fail`

Makes the branch log skimmable and produces a clean merge-commit summary.

## 8. Risks & mitigations

| Risk | Mitigation |
|---|---|
| Patchy test coverage in `playground/` means a pure-move can silently break behavior. | §6.5 Chrome-DevTools-MCP smoke pass after each file's splits; run `npm test` after every commit. |
| Moved code is referenced dynamically (e.g. string-keyed dispatch), so imports aren't enough. | Global grep of every moved symbol after each extraction; re-export from the original path until the dynamic reference is updated in the same commit. |
| `knip` false positives on handlers registered via event dispatch or MQTT topic strings. | Enumerate known dynamic entries in `knip.json` with a one-line reason per entry. Keep the list short and justified; if it grows, treat that as a signal to refactor toward static dispatch later. |
| `main.js` is 3556 lines of tangled concerns; extractions may reveal unexpected coupling. | Smallest commits possible, one module at a time, `npm test` green after each. If a module stubbornly won't extract cleanly, land a partial extraction (under cap but inelegant) rather than push it into another slice. |
| CI gate flipped to hard-fail catches a regression on a subsequent feature PR for reasons unrelated to file size. | The error output from `scripts/check-file-size.mjs` names the offending file and its line count — the remediation is obvious. |
